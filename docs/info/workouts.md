Workout data model and lifecycle
===============================

This document describes the data structures, storage, sync behavior and lifecycle for workouts used by the app. The source of truth for types and behaviour lives in the app package; see `packages/app/src/types/workout.ts`, `packages/app/src/contexts/WorkoutContext.tsx`, `packages/app/src/services/workoutStorage.ts`, and `packages/app/src/services/workoutApi.ts` for implementation details.

Key files
- `packages/app/src/types/workout.ts` — TypeScript interfaces for workouts, exercises, sets and context types.
- `packages/app/src/contexts/WorkoutContext.tsx` — Business logic: creating/updating/completing/discarding workouts; sync orchestration.
- `packages/app/src/services/workoutStorage.ts` — Local persistence (AsyncStorage) keys and helpers for active and pending workouts.
- `packages/app/src/services/workoutApi.ts` — Remote API calls used to save and read workouts.

Data structures
- Workout
  - `workoutId: string` — unique id for the workout (generated by the app using a timestamp + random suffix).
  - `templateId: string | null` — optional template reference (not required by core flow).
  - `startedAt: string` — ISO timestamp when workout started.
  - `completedAt: string | null` — ISO timestamp when workout finished; `null` while active.
  - `notes: string` — freeform notes text.
  - `tags: string[]` — list of tags attached to the workout.
  - `bodyWeight: number` — recorded body weight for the workout (units implied by app UI; validation exists in the UI).
  - `exercises: WorkoutExercise[]` — ordered list of exercises performed in the workout.
  - `createdAt: string` — ISO timestamp when the workout object was created.
  - `updatedAt: string` — ISO timestamp for last update to the workout object.
  - `syncStatus: 'pending' | 'synced'` — whether the workout has been synced to the backend.

- WorkoutExercise
  - `exerciseId: string` — unique id for this exercise instance inside the workout.
  - `savedExerciseId: string | null` — optional pointer to a SavedExercise entry.
  - `name: string` — exercise display name.
  - `order: number` — 1-based ordering of exercises in the workout.
  - `sets: WorkoutSet[]` — ordered list of sets for this exercise.

- WorkoutSet
  - `setId: string` — unique id for the set.
  - `order: number` — 1-based ordering of sets inside the exercise.
  - `reps: number | null` — number of repetitions (nullable while editing).
  - `weight: number | null` — numeric weight value (nullable while editing).
  - `weightUnit: 'kg' | 'lbs'` — unit for the weight value.

- SavedExercise (used by saved exercises feature)
  - `savedExerciseId: string`
  - `name: string`
  - `note: string`
  - `tags: string[]`
  - `archivedAt: string | null`
  - `createdAt: string`
  - `updatedAt: string`
  - `syncStatus: SyncStatus`

Lifecycle and UI rules
- Starting a workout: `WorkoutContext.startWorkout()` creates a new Workout object with `startedAt`, `createdAt`, `updatedAt` and `syncStatus: 'pending'`. See `packages/app/src/contexts/WorkoutContext.tsx`.
- Active workout: The active workout is saved to local storage via `saveActiveWorkout` on every change (`workoutStorage.saveActiveWorkout` uses AsyncStorage key `workout:active`). See `packages/app/src/services/workoutStorage.ts`.
- Completing a workout: `completeWorkout()` sets `completedAt` and `syncStatus: 'pending'`, saves the workout into the pending queue (`workout:pending`), clears the active workout, and dispatches the completed workout into the in-memory history. It then attempts to sync the completed workout to the API. See `WorkoutContext.completeWorkout`.
- Discarding a workout: `discardWorkout()` clears the active workout from storage and state and navigates back to Home.

Validation performed by UI before finishing a workout (important to keep in sync with any future backend validation):
- At least one exercise must be present.
- `bodyWeight` must be <= 1000 (UI check present in `ActiveWorkoutScreen.tsx`).
- Each exercise must have at least one set where `reps > 0` and `weight !== 0` (the app checks this before allowing completion).

Local storage and pending sync
- Active workout key: `workout:active` (single object).
- Pending workouts key: `workout:pending` (array of workouts awaiting server confirmation).
- When a workout is completed it is appended to the pending list (if not already present) and persisted with `savePendingWorkout`. The syncer will attempt to push pending workouts to the backend and remove those marked as saved. See `packages/app/src/services/workoutStorage.ts` and `WorkoutContext.syncWorkouts`.

Sync and API
- API surface: `packages/app/src/services/workoutApi.ts` exposes `saveWorkouts(workouts: Workout | Workout[])` and `getWorkouts(lastSK?: string)`.
  - `saveWorkouts` POSTs to `${baseUrl}/save` with authentication header and receives a result object with per-workout statuses.
  - `getWorkouts` reads workouts from `${baseUrl}/read` and supports pagination via `sk` query parameter. The app stores and uses `nextSk` on the `WorkoutState` to drive incremental reads.
- `syncStatus` is updated to `synced` for workouts that the server confirms. Failed syncs are tracked by `failedSyncIds` in the `WorkoutState`.

Server result shape (client-side expectation)
- `SaveWorkoutsResult` (see `workoutApi.ts`) includes `results: { workoutId: string; status: 'saved' | 'error' }[]` and an overall `success` flag. The client marks returned `workoutId`s as synced and removes them from local `workout:pending` storage.

IDs and ordering
- IDs are generated in the client using a `generateId()` helper (timestamp + random) in `WorkoutContext.tsx`.
- `order` fields for exercises and sets are 1-based and re-computed when items are removed to keep ordering contiguous.

Context API (operations exposed to UI)
- `startWorkout()` — create and set an active workout.
- `addExercise(name, savedExerciseId?)` — append a new exercise to the active workout.
- `removeExercise(exerciseId)` — remove exercise and re-order the remainder.
- `addSet(exerciseId)` / `removeSet(exerciseId, setId)` — manage sets for an exercise.
- `updateSet(exerciseId, setId, field, value)` — update `reps` or `weight` values.
- `updateExerciseName(exerciseId, name, savedExerciseId)` — rename exercise and associate a savedExerciseId.
- `updateNotes(notes)` / `updateTags(tags)` / `updateBodyWeight(weight)` — top-level workout fields.
- `completeWorkout()` — finalize workout and trigger sync.
- `discardWorkout()` — remove active workout without saving.
- `syncWorkouts([workouts])` — push pending or provided workouts to the backend.

Implementation notes and gotchas
- The UI enforces validation before allowing a workout to finish but the backend should also validate — do not rely only on client-side checks.
- `weightUnit` is stored per set and currently limited to `'kg' | 'lbs'`.
- `reps` and `weight` are nullable while the user edits a set; code must tolerate nulls when reading workouts.
- Pending syncs and active workout are both persisted locally so the app can recover state across restarts / offline usage.
- Pagination: `getWorkouts` returns `{ nextSk, workouts }` and `WorkoutContext` stores `nextSk` on the state. When restoring, the app reads the first page and merges local pending/work-in-progress workouts with API results.

Where to change field shapes
- Update `packages/app/src/types/workout.ts` first; then adjust `WorkoutContext.tsx`, UI screens (e.g. `ActiveWorkoutScreen.tsx`), and storage/ API helpers accordingly.

Verification
- To inspect runtime data, search the storage keys in `packages/app/src/services/workoutStorage.ts` and read AsyncStorage contents (debugging tools or RN AsyncStorage inspectors).
- To see sync behaviour, check `packages/app/src/services/workoutApi.ts` and the network requests originating from the app.
